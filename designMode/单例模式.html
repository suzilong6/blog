# 单例模式

# 简介

单例模式的定义是：保证一个类仅有一个实例，并提供一个访问它的全局访问点。

单例模式是一种常用的模式，有一些对象我们往往只需要一个，比如线程池、全局缓存、浏览器中的window对象等。在JavaScript开发中，单例模式的用途同样非常广泛。试想一下，当我们单击登录按钮的时候，页面中会出现一个登录浮窗，而这个登录浮窗是唯一的，无论单击多少次登录按钮，这个浮窗都只会被创建一次，那么这个登录浮窗就适合用单例模式来创建。

# 实现单例模式

function Singleton(name) {
  this.name = name;
}

Singleton.getInstance = function () {
  return Singleton.instance ? Singleton.instance : Singleton.instance = new Singleton(...arguments)
}

const singleton1 = Singleton.getInstance('jack')
const singleton2 = Singleton.getInstance('tom')
console.log(singleton1 === singleton2); // true

# 透明的单例模式

我们现在的目标是实现一个“透明”的单例类，用户从这个类中创建对象的时候，可以像使用其他任何普通类一样。

const Singleton = (function () {
  let instance;
  return function (name) {
    if (instance) return instance;
    this.name = name;
    return instance = this;
  }
})()

const singleton1 = new Singleton("tom");
const singleton2 = new Singleton("jack");
console.log(singleton1 === singleton2); // true

虽然现在完成了一个透明的单例类的编写，但它同样有一些缺点。为了把instance封装起来，我们使用了自执行的匿名函数和闭包，并且让这个匿名函数返回真正的Singleton构造方法，这增加了一些程序的复杂度，阅读起来也不是很舒服。

让我看到构造函数，它实际上执行了两个逻辑，一个是实现单例的逻辑，一个是构造函数本身的逻辑，这样就违背了单一职责原则。

# 用代理实现单例模式

现在我们通过引入代理类的方式，来解决上面提到的问题。

function User(name) {
  this.name = name
}

const proxyCreateUser = (function () {
  let instance
  return function () {
    return instance ? instance : instance = new User(...arguments)
  }
})()

const user1 = proxyCreateUser('tom')
const user2 = proxyCreateUser('jack')
console.log(user1 === user2) // true

# JavaScript中的单例模式

前面提到的几种单例模式的实现，更多的是接近传统面向对象语言中的实现，单例对象从“类”中创建而来。在以类为中心的语言中，这是很自然的做法。比如在Java中，如果需要某个对象，就必须先定义一个类，对象总是从类中创建而来的。

但JavaScript其实是一门无类（class-free）语言，也正因为如此，生搬单例模式的概念并无意义。在JavaScript中创建对象的方法非常简单，既然我们只需要一个“唯一”的对象，为什么要为它先创建一个“类”呢？这无异于穿棉衣洗澡，传统的单例模式实现在JavaScript中并不适用。

单例模式的核心是确保只有一个实例，并提供全局访问。

全局变量不是单例模式，但在JavaScript开发中，我们经常会把全局变量当成单例来使用。例如：

let a = {}
1

当用这种方式创建对象a时，对象a确实是独一无二的。如果a变量被声明在全局作用域下，则我们可以在代码中的任何位置使用这个变量，全局变量提供给全局访问是理所当然的。这样就满足了单例模式的两个条件。

但是全局变量存在很多问题，它很容易造成命名空间污染。

# 惰性单例

惰性单例指的是在需要的时候才创建对象实例。惰性单例是单例模式的重点，这种技术在实际开发中非常有用，有用的程度可能超出了我们的想象。

假设现在有一个需求，网站的的登陆弹窗功能，在同一时刻，只会出现一个登录弹窗，因此使用单例模式非常适合。但是有些用户进入网站只是随便看看，可能并不需要登录，因此不需要创建登录弹窗，这时我们可以使用惰性单例模式，只有当用户点击登录按钮才创建这个弹窗。

const showMessageBox = (function () {
  let messageBox
  return function () {
    if (messageBox) return
    messageBox = document.createElement('div')
    messageBox.innerHTML = '我是登录弹窗'
    document.body.appendChild(messageBox)
  }
})()

document.querySelector('.login-but').addEventListener('click', () => {
  showMessageBox()
})

但是我们发现它还有如下一些问题。

  1. 这段代码仍然是违反单一职责原则的，创建对象和管理单例的逻辑都放在showMessageBox函数内部。
  2. 如果我们下次需要创建页面中唯一的iframe，或者script标签，用来跨域请求数据，就必须得如法炮制，把showMessageBox函数几乎照抄一遍：

下面我们再来优化一下：

我们需要将不变的部分和变化的部分分离开来：不变的部分是管理单例的逻辑，变化的部分是创建弹窗的业务逻辑。

const getSingle = function (fn) {
  let result;
  return function () {
    return result || (result = fn.apply(this, arguments));
  }
};

const showMessageBox = function () {
  const messageBox = document.createElement('div')
  messageBox.innerHTML = '我是登录弹窗'
  document.body.appendChild(messageBox)
  return messageBox
}

const singleShowMessageBox = getSingle(showMessageBox)

document.querySelector('.login-but').addEventListener('click', () => {
  singleShowMessageBox()
})
